# Review-artifact persistence: ephemeral-local working state, committed decisions only

Status: Approved

## Problem

The review engine (`#ai-review-engine`) has two applications whose artifacts are persisted inconsistently, and one of them (`#ai-review-board`) commits low-signal, high-churn, never-validated machine output into the repo:

- **Board** (`#ai-review-board`): designed to commit a canonical issue store under `reviews/` (per `issue-format.md`), with cross-round reconciliation, `baselineCommit` chaining, and a store lint framed as a CI/pre-commit gate. In practice `reviews/` was never populated here, and a committed store of AI-raised-but-unvalidated findings would (a) conflict across parallel branches, (b) churn history, (c) duplicate the team's real tracker, which the board already treats as the validated sink ("a triage layer on top of the tracker").
- **Instruction review** (`#ai-instruction-review`): commits the full rolling backlog `docs/future-work/proposed-instruction-rules.md` (~30KB). It runs a **full audit every round**, so the open-proposal list (and its drafts) is *regenerated each round regardless of the backlog* — most of that committed file is regenerable, unvalidated machine output. The only durable signal is the **decisions**: why a proposal was rejected/folded, and what is deferred until when. The drafts are not durable — they live in git history and are re-derived on demand.

## Principle (the rule this spec encodes)

Three tiers decide where a review artifact lives:

1. **Working / reasoning state** → local ephemeral (`.agentsmith/`, gitignored). Per-run reviewer outputs, verify transcripts, scorecards, the open issue/proposal queue.
2. **Validated output** → its owning system. Board issues → the team's external tracker (via elevate/`/review-promote`). Adopted rules → `instructions/` (a commit). Adoption/elevation *is* the validation.
3. **Decision memory** → committed, lean — **only** when the subject is global/shared and re-deriving the decision each round is expensive. Instruction-review: yes (subject is the global instruction set). Board: no (findings are branch-scoped and cheap to re-verify).

The deciding test is not "valuable vs. transient" but **"does an external system already own this subject's truth?"** Code issues → the tracker owns them → don't commit. Rule decisions → nothing else owns them → the repo does.

## Change A — Board: local-ephemeral store + elevate + baseline gate

A1. **The whole board — store *and* config — moves out of git.** The canonical issue/epic/round **store** and the **`config.yaml`** (active roles + gating, plus the `instruction-review.participants` override) both move under the **persistent-but-local** directory `.agentsmith/review-board/` (gitignored via the existing `.agentsmith/` rule): `.agentsmith/review-board/config.yaml` + the store tree beside it. It persists across rounds on one machine (so local reconciliation and the `baselineCommit` chain work) but is never committed. This is distinct from `.agentsmith/tmp/review-board/<round-id>/`, which stays per-run scratch. **`reviews/` is removed entirely** — nothing under it remains committed.
   *Rationale (user revision, overrides round-1 B7):* config only has meaning alongside the generated artifacts; with the board itself per-machine and local, publicly committing its config while the findings it gates are not shared is incoherent. The default config still **bootstraps from the tool** when absent (the board SKILL already carries the default), so a fresh clone regenerates it — no sharing needed. A team that wants shared gating customizes its own copy; consistency of the *default* participant/role set is preserved because that default is baked into the SKILL, not the committed file.

A2. **The tracker is the only durable, shared sink.** Validated findings are elevated to the team's external tracker (`/review-promote` records the tracker ref; the future "tracker API bridge" automates it). Nothing AI-raised-but-unvalidated is publicly persisted. The local `promoted` marker remains only to stop re-raising; the tracker is the record of truth.

A3. **Baseline confirmation gate.** At board setup, ask the user to confirm `baselineCommit` (mirrors the `#ai-preflight` recommended-default pattern). The recommended default is computed first, then offered; the cases (resolves B3):
   - **(i-feature) automated, feature branch** *(recommended)* — `merge-base(commit, default)` (pure git, no stored state, always derivable).
   - **(i-main) automated, recurring `main` round** *(recommended)* — the **normal main-branch case**: baseline = the SHA of the **prior `main` round** recorded in the local store (i.e. "current `main` HEAD vs the last `main` review round"). This is the default whenever a prior `main` round exists locally.
   - **(i-bootstrap) first-ever `main` round** *(recommended when no prior `main` round exists)* — `baselineCommit = default HEAD`, full sweep, labeled as bootstrap (distinct from (ii): genuine first round, not a lost store).
   - **(ii) fresh full sweep** — zero carry-forward; recommended when the store exists but the user wants a clean pass.
   - **(iii) user-provided commit hash** — for pinning an explicit baseline.
   **Absent-store routing (resolves B10; feature-branch case per N5):** for a **feature-branch** round, (i-feature) `merge-base` is store-independent and stays the auto-recommended default even with an absent store. The ambiguity is only for a **`main`** round: once the store is gone the agent *cannot* know whether a prior `main` round existed — that fact lived in the store. So on an absent/empty store for a `main` round the agent does **not** try to infer; it presents (i-bootstrap), (ii), and (iii) together and asks the user to choose (no auto-recommendation), with one line noting prior `fixed`/`promoted` continuity is only recoverable via the tracker. Missing local state never silently breaks; it always surfaces the choice.

A4. **Absent-store behavior (resolves B2; reconciled with A3 per B11).** With a missing or empty `.agentsmith/review-board/` (fresh clone, wiped machine) there are **no prior open issues to reconcile against — zero carry-forward is unavoidable** (there is nothing to carry). The agent does not *unilaterally* pick a mode: the **baseline is resolved through the A3 gate**, which on an absent store offers (i-bootstrap)/(ii)/(iii) and asks. Every absent-store option is necessarily a from-scratch pass (they differ only in baseline), so "no carry-forward" holds regardless of the choice. Known, accepted trade-off to state in the rule and `issue-format.md`: issues `promoted`/`fixed` on another machine are **not recoverable** from the local store; their durable record is the **tracker** (promoted) or **git history** (fixed commit). Cross-machine continuity is the tracker's job, not the store's.

A5. **Lint reframed (resolves B1; invocation per B12).** `lint.mjs` stays but is repointed at `.agentsmith/review-board/` and reframed from "CI/pre-commit gate on committed `reviews/`" to a **local integrity check** the round runs (SKILL step 4) against the local store. The invocation changes from `node .claude/skills/review-board/lint.mjs reviews` to **`node .claude/skills/review-board/lint.mjs .agentsmith/review-board`**; the CI/pre-commit framing in `issue-format.md §Validation` and `SKILL.md` is **struck** (no committed store path to gate). `lint.mjs` still validates the local `.agentsmith/review-board/config.yaml` shape if desired, but is not a commit gate (there is nothing committed left to gate).

A6. **Store invariants rewritten to local scope (resolves B4).** `issue-format.md §The store` / §Compositional id assert "No agent deletes store files — git history is the archive" and "Ids are never reused, so `relatedIssues` links stay valid forever." These are rewritten to **local-lifetime** guarantees: ids stay unique and links stay valid *within a machine's local store*; the cross-machine, forever guarantee moves to the tracker for promoted issues. The `#ai-review-board` rule's "never agent-deleted committed store" clause is replaced accordingly (no committed store to be the archive).
   **`relatedIssues` referential lint stays an error (resolves C11):** within a live local store nothing deletes (promoted issues stay in the `promoted/` partition, frozen — not removed), so refs never dangle; a store loss is **atomic** (the whole `.agentsmith/review-board/` directory disappears together), leaving an *empty* store and a forced full sweep with no carried issues to reference anything. There is no partial-absence state, so the lint's "`relatedIssues[].issueId` must resolve" check remains a hard error — it can only fire on a genuine bug, never on the wipe scenario.

## Change B — Instruction review: ephemeral queue + drafts, committed decisions only

Drafted-but-unadopted rules are **unvalidated machine output** — the same category we are removing from the board. They are not made public; only **decisions** persist, so future rounds neither re-litigate nor re-raise them. This supersedes the spec-reviewer's B5 (commit the drafts): B5's "lost craft" is not lost — the old backlog's drafts remain in **git history** after the file is deleted, and a full audit **re-derives and re-vets** a draft on demand when a deferred rule's condition is met. (B9, the "what is a vetted draft" definition, is therefore moot — nothing draft-based is committed.)

B1. **Everything except decisions is ephemeral.** The open-proposal queue (ready *and* conditional, **with their drafts**), the dimension scorecard, and the mechanical nits are written only to `.agentsmith/tmp/instruction-review/<round-id>/` and presented — never committed. The full audit regenerates them each round.

B2. **The single committed output is a lean decisions log: `docs/instruction-rules-decisions.md`.** An ADR-style file, one line per closed judgment, no draft blocks:
   - `#tag — rejected: <reason>` (e.g. "`#git-commit-signing` — rejected: generic, no repo anchor")
   - `#tag — folded into <X>: <reason>` (e.g. "`#swe-regression-tests` — folded into #swe-testing: tests-committed already covers it")
   - `#tag — deferred: <condition to adopt> (→ <targetFile>, <role>)` (e.g. "`#swe-ci` — deferred: adopt when the project runs CI (→ swe.md, swe)") — the deferral decision, its trigger, and the **placement hint** persist (C5: cheap to keep, non-obvious to re-derive); the **draft block does not** (a future round re-drafts when the condition holds).
   This stops re-litigation (rejected/folded) and premature re-raising (deferred-conditional) without publishing any unadopted rule text. **Uniqueness (C17):** the log holds at most one entry per `#tag`; the instruction-review SKILL checks for an existing entry before appending (a later decision on the same tag updates the line, never duplicates it) — otherwise duplicates would erode the "stops re-litigation" guarantee.

B3. **Migration of the current backlog.** From `docs/future-work/proposed-instruction-rules.md`: extract the **decisions** into `docs/instruction-rules-decisions.md` — the rejected/folded notes (one line each) and the conditional B/C/D bands collapsed to `#tag — deferred: <condition> (→ <targetFile>, <role>)` lines (placement hints preserved per B2/C14; **drafts dropped**, recoverable from git history and regenerated on demand). Drop the ready-portable open proposals (re-found by the next audit; adopt them now if wanted). Then remove the old file. (Resolved Open Decision 2: one file, `docs/instruction-rules-decisions.md`; user may rename, but a concrete value is required since `proposal-format.md`/`SKILL.md` hardcode the old path.)

## Change C — Instruction-check: no change

Already ephemeral by design (single-agent, no persistence). Documented here only to record that it stays console/`.agentsmith`, never committed.

## Change D — Instruction-review triage step (the human gate over the three sinks)

The round gains a final **triage** step (after reduce, replacing the passive "present"): each verified proposal is shown — its kind, tag, gap/reasoning, and proposed wording — and the user gives it a **disposition**, which routes it to exactly one of the three sinks. This is the instruction-review analog of the board's `review-promote` (human validation is the gate), and it operationalizes the persistence model: nothing is committed or adopted without an explicit human disposition.

D1. **The four dispositions:**
   - **(i) reject** → one line appended to the committed decisions log (`#tag — rejected: <reason>`); never re-raised.
   - **(ii) decide later** → parked in a per-machine `.agentsmith/instruction-review/parked.md` (resolved per C7 — persistent so it survives the session; gitignored under `.agentsmith/`, not committed, not adopted). **Re-surfacing (C9):** at the start of each round's triage the agent **reads `parked.md`, de-duplicates against the fresh queue, and prepends** the remainder, so parked proposals are offered again; an entry is dropped when its **exact tag** now appears in `instructions/ownership.yaml` or the decisions log (C16: tag-equality only — gap-*subsumption* by a different adopted tag is not auto-detected; such a stale entry simply re-surfaces and the user re-disposes it). This is what makes "later" actually return — the audit alone regenerates the queue from scratch and would not otherwise carry parked items. **In-session bookkeeping (resolves C12):** the moment a proposal is dispositioned this session — whether it came from the fresh queue or from `parked.md` — its `parked.md` entry (if any) is removed immediately, and re-parking writes a fresh entry. A proposal is therefore never offered twice in one triage pass, and `parked.md` only ever holds still-undecided items.
   - **(iii) adopt** → **guided adoption, not a verbatim paste** (see D2).
   - **(iv) other** → free text; the agent interprets and acts (e.g. "adopt but reword X", "merge with proposal N", "fold into #tag").

D2. **Adoption (iii) runs the full change discipline, branched by proposal kind** — the round may now *write to `instructions/`*, but only through the same mechanics a human adoption would use, never a blind append:
   - `new-rule` → add the rule to its `targetFile`/section **and** add its `ownership.yaml` row (omitting it fails the coverage lint → red `npm test`), then regenerate `AGENTS.md` and run `npm test`; show the diff.
   - `strengthen` → edit the existing rule in place (it is an *edit*, not an add).
   - `rehome` → move the rule to `proposedFile`. **Cross-boundary guard (C4):** a move across the core/bundle line can trip the lean-split lint (a core rule referencing a bundle-only tag). The adoption runs `npm test`; on failure it does **not** force the move — it blocks that adoption, reports the lint error, and offers the user a redirect (different `proposedFile`) or defer, in-session.
   - `reowner` → change only the `ownership.yaml` row. **Validity guard (C3):** the reduce stage already rejects a `proposedOwner` that is not a declared role/marker; adoption does not re-trust blindly — it still runs `npm test`, whose coverage lint catches an unresolved owner, an orphaned tag, or a role left with zero tags, and blocks the adoption if any fires.
   Adoptions are applied **one proposal at a time**, each followed by `npm test` (coverage lint + dangling-tag + build), so a failure isolates to that proposal. Every adoption must leave `npm test` green before the round reports it done (#swe-done). **On any failed adoption** (lint, build, or test): block *that* proposal, surface the exact failure, and offer the user a redirect / defer / reject in-session; the round continues with the remaining proposals.
   **Revert mechanism (C8):** an adoption writes only git-tracked source files (`targetFile`, `instructions/ownership.yaml`); `AGENTS.md` is a gitignored build artifact, regenerated not reverted. On failure, revert with `git restore <targetFile> instructions/ownership.yaml` then rebuild, so no half-write survives. The decisions log is **not** in the revert scope (C18): a reject/defer append is an already-made decision, append-only, not speculative — a later failed adoption never unwinds it. **Guard (C13):** before the first adoption, check those files for pre-existing *unrelated* uncommitted edits — a `git restore` would discard them; if present, **ask the user to stash or commit first** before proceeding with adoptions.

D3. **Charter change.** `#ai-instruction-review` is currently *"proposes only — never edits instruction sources; adoption is a separate human act."* It relaxes to: *"proposes; in the triage step adopts only the proposals the user explicitly accepts, each through the normal change discipline (#swe-done)."* The decisions log and the propose/verify/reduce pipeline are unchanged; only the final step gains adoption authority, gated per-proposal by the human.

D4. **Interaction discipline.** Triage is **optional** and **batched**, respecting `#ai-preflight`: offer it after reduce; in non-stop/batch mode present all cards in one `AskUserQuestion` pass (multi-select per proposal) rather than N sequential round-trips. **Heterogeneous dispositions (C15):** `reject`/`decide-later`/`adopt` are selectable in the batch pass; a proposal marked `other` (free text) or an `adopt` whose guided sub-steps need input falls back to a short sequential follow-up after the batch. A user who declines triage gets the old passive present (proposals listed, adoption left for later).

3. **Drafts ephemeral, not committed** — **resolved: drafts are unvalidated output and stay ephemeral** (reverses reviewer B5; B9 moot). Only decisions persist. Lost-craft objection answered by git history + on-demand re-derivation.
4. **Lint as CI gate** — **resolved: accept the loss** [A5]; the store is local, the tracker is the shared gate. (Reversible later via the deferred tracker-API-bridge.)
5. **No-tracker repos** — **resolved: tracker-optional** — no tracker → local store is the working record; `promoted` is either skipped (issues go straight to `fixed`/`deprecated`) or `promotedTo` records a local note. Concrete lint relaxation (closes review nit final-b): **`promotedTo` is required for a `promoted` issue only when a tracker is configured in `.agentsmith/review-board/config.yaml`; with no tracker configured, lint treats a missing `promotedTo` as valid** (and `promoted` is discouraged in favor of a closing status). With a tracker, elevation is the durable sink as in A2.
